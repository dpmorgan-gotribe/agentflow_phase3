Root cause:
  feature-graph.ts had reviewer-driven retry routing (bug-109 at
  lines 1383-1627) but NO security-analog. The dispatch loop's per-task
  retry fallback re-dispatched the same agent against unchanged code.

Three-part fix shipped (mirrors bug-109 shape):

  Part A — packages/orchestrator-contracts/src/security.ts
    No schema change needed. SecurityAgentOutput.findings[].retryTarget
    already exists as SecurityRetryAgent enum (backend-builder |
    web-frontend-builder | mobile-frontend-builder | tester). The schema
    was correct; only orchestrator routing was missing.

  Part B — orchestrator/src/invoke-agent.ts
    - Added SecurityAgentOutputType import (alongside ReviewerOutputType)
    - Added securityOutput? field on InvokeAgentResult interface
    - Added parse block: when agent === "security", run
      SecurityAgentOutputSchema.safeParse on extracted JSON +
      capture into securityOutput for feature-graph to consume
    - Returns securityOutput in captureReturn shape when defined

  Part C — orchestrator/src/feature-graph.ts
    - Added SecurityAgentOutputType import (alongside ReviewerOutputType)
    - Added securityOutput? to InvokeAgentResult interface (with bug-007
      doc comment mirroring bug-109's reviewerOutput doc)
    - Added security-driven routing block (~lines 1631-1830) AFTER the
      existing reviewer routing block + BEFORE the bug-121 tester block.
      Algorithm:
        1. On agent==="security" && securityOutput && verdict !== "approved":
           - "blocked" → fail feature immediately (P0 finding present)
           - "needs-revision" → routing loop bounded by TASK_RETRY_CAP:
             a. Group findings[] by retryTarget agent name
             b. For each named agent: re-dispatch with HARD CONSTRAINT
                bug-007 envelope inlining the findings (severity +
                OWASP + CWE + file + line + title + description + fix)
             c. Commit retry-target's work so security re-run sees
                fresh code
             d. Re-dispatch security to re-validate
             e. "approved" → break + continue agent_sequence
             f. "blocked" → fail feature
             g. else loop
        2. On retry-cap exhaustion → security-cap-exhausted (bug-007)
           abort reason

  Part D — orchestrator/tests/feature-graph.test.ts
    3 new test cases appended in describe("runFeature — security-driven
    retry routing (bug-007)") block:
      1. routes needs-revision → builder retry with HARD CONSTRAINT →
         re-runs security → approved
      2. fails feature on blocked verdict WITHOUT re-dispatching builder
         (P0 secret-exposure case)
      3. aggregates findings by retryTarget agent (multiple findings →
         single builder dispatch carrying all findings in one retry
         envelope)
