# 027: Shared Packages Skeleton

## Invocation Point (refactor-003)

Invoked from `/new-project` step 5b (task 018b) immediately after task 026 creates the monorepo root. This runs at project-bootstrap time because:

Architect (task 020, post-design) later reads the kit's `package.json.version` (at signoff time the version is locked) and adds vendor-specific dependencies to `apps/*` but does NOT modify the package skeletons themselves.

## What This Task Produces

Working starter code for four shared packages that eliminate cross-app duplication. Per refactor-001, the previously-separate `@repo/tokens` and `@repo/ui` are **replaced by a single `@repo/ui-kit`** scaffold — this task creates the empty kit skeleton; task 024 (`/stylesheet`) fills it in.

## Why This Scope (per refactor-001)

- **One kit, not two packages.** `@repo/tokens` + `@repo/ui` collapse into `@repo/ui-kit` because tokens, stylesheet, primitives, patterns, and layouts are one system with one public API. Two packages invite drift and deep-import footguns.
- **Leaf package.** `@repo/ui-kit` imports from nothing internal to the workspace. It is the root of the dependency tree for all front-end apps.
- **Scaffold here, fill there.** This task creates the empty directory structure and `package.json` with correct dependencies — `/stylesheet` (024) runs later in the pipeline to populate tokens.json, primitives, patterns, layouts, Storybook, etc.
- **022b artifacts land inside the kit.** `ESLint plugin`, `validate-consumer.ts`, and `CONTRACT.md` live at `packages/ui-kit/eslint-plugin/`, `packages/ui-kit/scripts/`, and `packages/ui-kit/CONTRACT.md` respectively.
  - **CONTRACT.md** is project-invariant (same rules every project). It's written **here with real content** — sourced from a template at `.claude/templates/ui-kit-contract.md` (owned by task 022b). This way downstream agents can read the contract from day zero, not wait for 024.
  - **ESLint plugin** (`eslint-plugin/`) and **validate-consumer.ts** are skeleton only here; task 024 writes their actual implementation because the rules' regex patterns depend on the kit's public-barrel contents (which only exist after 024 runs).

**pnpm-workspace.yaml glob requirement.** The ESLint plugin is a workspace package nested inside `packages/ui-kit/`. The default `packages/*` glob would NOT resolve it. Task 026 (Turborepo + pnpm workspace scaffold) must use one of these patterns in `pnpm-workspace.yaml`:

```yaml
# Option A — recursive glob (simplest)
packages:
  - "packages/*"
  - "packages/ui-kit/eslint-plugin"

# Option B — explicit nested
packages:
  - "packages/*"
  - "packages/ui-kit/eslint-plugin"
  - "apps/*"
```

If the workspace glob is wrong, `pnpm install` will not register `@repo/eslint-plugin-ui-kit-contract` and consumer ESLint configs will fail to resolve it. This task's acceptance verifies the plugin is listed in `pnpm list --depth -1 --json` across the workspace.

### Platform variant pattern (unchanged concept; new location)

React Native variants live **inside** `@repo/ui-kit/src/primitives/` alongside the web version, not in a separate package:

```
packages/ui-kit/src/primitives/button/
├── Button.tsx           # default (web) — rendered via React DOM
├── Button.native.tsx    # React Native Paper equivalent — same props surface
├── Button.variants.ts
├── Button.stories.tsx
└── index.ts             # re-exports Button (Metro picks .native.tsx for RN builds)
```

Metro (the RN bundler) automatically resolves `.native.tsx` over `.tsx` for native targets; webpack/vite resolve `.tsx`. The props API is identical. `@repo/ui-kit` is the one import source for both web and native consumers.
